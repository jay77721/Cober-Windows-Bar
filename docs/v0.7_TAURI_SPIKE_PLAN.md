# v0.7 Tauri Spike Scope Freeze

## Baseline

- v0.6 Mock Provider SDK alignment is closed at `92f3e01 test: harden provider alignment coverage`.
- The current provider runtime remains mock-only and browser-local.
- The canonical event path remains `Provider -> Event Bus -> Store -> Resolver -> UI`.

## Route Decision

v0.7 is the Tauri shell/runtime/IPC boundary planning and spike scope-freeze phase. It is not the first real-provider phase. First real provider work moves after the Tauri boundary is proven, likely v0.8 or later.

Mock-provider behavior hardening remains targeted support work only when it protects the Tauri fixture path. It is not the main v0.7 route.

## Spike Goal

Prove that the existing web hub can run behind a desktop shell/runtime boundary and receive canonical mock or fixture HubEvents through a narrow IPC contract without bypassing the Event Bus, Store, Resolver, or UI boundary.

## Allowed Scope

- Define the Tauri shell viability questions.
- Define the runtime and IPC boundary shape.
- Define how mock or fixture canonical HubEvents cross the boundary.
- Define diagnostics for unavailable or malformed native boundary data.
- Define planning-level success criteria before native implementation starts.

## Explicit Non-Goals

- Real providers.
- Windows APIs, media sessions, notification readers, file watchers, or system metrics.
- Broad Rust module design.
- Production packaging polish, signing, updater, or installer work.
- UI redesign.
- Store, Resolver, or ProviderRegistry behavior changes.
- Package, script, or asset changes unless separately approved.

## Minimum Success Criteria

- The route is reconciled: v0.6 is provider alignment, v0.7 is Tauri boundary planning/spike, and first real provider follows after the boundary is proven.
- The spike has a narrow shell/runtime/IPC goal using mock or fixture canonical HubEvents.
- The IPC boundary preserves normalized `HubEvent` payloads and keeps UI agnostic to mock, IPC, or future Windows sources.
- The existing event path remains authoritative: Event Bus -> Store -> Resolver -> UI.
- Failure and diagnostics are defined at boundary level without adding real provider behavior.
- Non-goals are explicit enough to stop Tauri scope creep into real providers or broad Rust architecture.

## Stop Conditions

- The work requires real Windows provider data.
- The work requires Store, Resolver, or UI behavior changes.
- The work requires package/script/assets changes outside a separately approved setup task.
- The work expands into broad Rust architecture before the shell/runtime/IPC boundary is frozen.
